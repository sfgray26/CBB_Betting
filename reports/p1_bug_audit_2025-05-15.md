## The 5 P1 Bugs — Detailed Analysis

### Bug 1: Roster Optimize Endpoint Uses Wrong Solver ⭐ CRITICAL

**Location:** `backend/routers/fantasy.py:3353-3428` (optimize_roster endpoint)

**Problem:** The production endpoint uses a simple greedy slot-filling algorithm that assigns players by descending score without considering positional scarcity or multi-position flexibility:

```python
# Current (broken) approach in fantasy.py
player_data.sort(key=lambda x: x["lineup_score"], reverse=True)  # Sort by score only
# Then fill slots greedily in priority order
```

**What Exists But Isn't Used:** `backend/fantasy_baseball/lineup_constraint_solver.py` — a fully-tested OR-Tools CP-SAT solver that:
- Solves as an Integer Linear Programming problem (mathematically optimal)
- Respects positional scarcity (C and SS are scarcest, filled first)
- Handles multi-position players optimally
- Provides natural-position bonuses
- Falls back to scarcity-aware greedy if OR-Tools unavailable

**Impact:** Wrong lineup slots assigned, leaving better-scoring players on bench while inferior players start due to poor slot allocation.

**Fix:** Replace the endpoint's greedy logic with a call to `LineupConstraintSolver.solve()`.

### Bug 2: Inverted Implied Runs Sign ⚠️ HIGH

**Location:** `backend/fantasy_baseball/daily_lineup_optimizer.py:420-436`

**Problem:** The `_implied_runs()` function has inverted sign logic for home favorites. When home team is favored (negative spread like -1.5), they get **fewer** implied runs instead of more:

```python
# Current (broken) code
home_runs = (total + spread_home) / 2.0  # When spread_home = -1.5, this SUBTRACTS

# Example: total=9, spread_home=-1.5 (home favored)
# Current: home_runs = (9 + (-1.5)) / 2 = 3.75 runs (WRONG - underdog level)
# Should be: home_runs = (9 - (-1.5)) / 2 = 5.25 runs (favored team scores more)
```

The docstring even shows the correct formula `(total - spread_home) / 2 + spread_home` but the code uses `+ spread_home` instead of `- spread_home`.

**Impact:** Batter rankings and streaming pitcher recommendations are backwards — players facing favored home teams get undervalued, players facing home underdogs get overvalued.

**Fix:** Change line 431 to:
```python
home_runs = (total - spread_home) / 2.0  # Note the minus sign
```

### Bug 3: Silent Empty Roster on Missing Count Field ⚠️ HIGH

**Location:** `backend/fantasy_baseball/yahoo_client_resilient.py:693-701`

**Problem:** The `get_roster()` method silently returns an empty list if Yahoo's payload omits the `count` field:

```python
count = int(players_raw.get("count", 0))  # Defaults to 0 if missing

for i in range(count):  # Loop never executes if count=0
    # ... process players

return list(players_by_key.values())  # Returns [] if count was missing
```

Under Yahoo API shape variance, if the `count` field is missing or malformed, the entire roster appears empty without any warning or error.

**Impact:** Cascading "no roster" failures — lineup optimization, matchup analysis, and waiver recommendations all break silently.

**Fix Options:**
1. **Defensive:** Try to infer count from array length if field missing
2. **Fail-fast:** Raise explicit error if count is missing and players data exists
3. **Hybrid:** Log warning, attempt recovery from array structure

### Bug 4: Disabled Pitcher Handedness Signal ⚠️ MEDIUM

**Location:** `backend/services/constraint_helpers.py:204-251` (lookup_opposing_sp)

**Problem:** The pitcher handedness signal (claimed as 35% of matchup score weight) never activates because:

```python
# Line 240 - tries to get handedness from probable pitcher data
sp_handedness=pp.get("handedness"),  # "L" or "R"

# But if handedness isn't populated in the probable_pitchers data:
return OpposingSPInfo(
    sp_name=None,
    sp_handedness=None,  # Always None when no pitcher found
    ...
)
```

The signal flows through to `OpposingSPInfo.sp_handedness` but downstream consumers find `None` and skip platoon split calculations.

**Impact:** Matchup scores are less discriminating than advertised — platoon advantages (LHB vs RHP, RHB vs LHP) aren't factored into batter rankings.

**Fix:** Ensure handedness is populated in probable pitcher ingestion, or fetch it on-demand from a reliable source (Statcast, FanGraphs) when missing.

### Bug 5: Unsafe Live Projection Fallback — PARTIALLY CONFIRMED

**Location:** HERMES.md cites `projection_assembly_service.py:503-530` but this file doesn't exist in the codebase.

**Investigation:** Searching for `ilike` patterns found:
- `backend/services/daily_ingestion.py:5314`: `.filter(PlayerProjection.player_name.ilike(p.get("name", "")))`
- `backend/fantasy_baseball/player_board.py:1134`: `.ilike(f"%{name}%")`
- `backend/fantasy_baseball/mcmc_calibration.py:109`: `.ilike(player_name)`

The daily_ingestion.py pattern is problematic — if `p.get("name", "")` returns empty string, the query becomes `ilike('')` which could match empty strings or behave unpredictably across databases.

**Impact:** Potential for wrong projections to be attached to Yahoo-only players when name matching fails.

**Fix:** Add guard clause before ilike:
```python
name = p.get("name", "").strip()
if name:
    query = query.filter(PlayerProjection.player_name.ilike(name))
```
